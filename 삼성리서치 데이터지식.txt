# GenOS 온프레미스 데이터 및 인프라 명세서

## 1. 데이터 소스 및 접근 방식 (Data Source Mapping)
이 시스템은 외부망 접속이 불가능하므로, 중계 서버를 통해 내부로 반입된 데이터만 활용한다.

| 데이터 분류 | 상세 항목 | 소스/형식 | 접근 방식 (Flowise Node) |
| :--- | :--- | :--- | :--- |
| **VDB (비정형)** | 국내 뉴스 | mpop 아너스넷 (기존 RDB) | `Vector Store Retriever` (Embedding 후 검색) |
| **VDB (비정형)** | 해외 뉴스 | Factset | `Vector Store Retriever` |
| **VDB (비정형)** | 해외 SNS | Stockwits | `Vector Store Retriever` (감성 분석 등 활용) |
| **VDB (비정형)** | 국내 공시 | Dart API (HTML) | `Vector Store Retriever` (HTML 파싱 후 Chunking) |
| **VDB (비정형)** | 리포트 | PDF (국문) | `Vector Store Retriever` (PDF Loader 활용) |
| **VDB (비정형)** | IR 자료 | 애널리스트 공용 폴더 | `Vector Store Retriever` (이미지/차트 포함 시 Qwen3-VL 활용) |
| **VDB (비정형)** | 해외 실적발표 | Factset Transcript | `Vector Store Retriever` |
| **RDB (정형)** | 국내 주가/재무 | 에프엔가이드 | `SQL Agent` 또는 `Custom API Tool` |
| **RDB (정형)** | 해외 주가/재무 | Factset (신규) | `SQL Agent` 또는 `Custom API Tool` |
| **RDB (정형)** | 개인 데이터 | Excel 시계열 (신규) | `SQL Agent` (업로드 후 테이블화 가정) |
| **RDB (정형)** | 매크로 지표 | Macrobond (신규) | `SQL Agent` |

## 2. 하드웨어 리소스 할당 가이드 (Resource Allocation)
* **총 자원:** H200 (141GB VRAM) x 16장
* **배포 전략:**
  * `gpt-oss-120b`: 4~8장 할당 (vLLM 또는 TGI 서빙) - 빠른 추론 속도 필요 시
  * `Qwen3-VL-235B`: 8장 이상 할당 (멀티모달 처리 시 메모리 요구량 큼)

## 3. 기능 구현 패턴 (Functional Patterns)

### A. AI Chat (Hybrid Retrieval Agent) - **[가장 중요/메인 기능]**
사용자의 자연어 질문을 분석하여, 실시간 데이터 조회(SQL)와 문맥 검색(RAG)을 스스로 판단하여 수행하는 중앙 허브 기능.

* **주요 사용 케이스:**
    1.  **조건 검색 (Filtering/Sorting):** "최근 수익률 상위 5개 ETF 보여줘", "수수료 0.5% 이하인 반도체 펀드 찾아줘"
    2.  **비교 분석 (Comparison):** "A기업 현재 주가랑 컨센서스(목표주가) 비교해줘"
    3.  **복합 질문:** "최근 2차전지 섹터 뉴스 요약해주고(VDB), 관련 대장주 3개 PER 알려줘(RDB)"

* **데이터 흐름 및 노드 구성 전략 (Flowise):**
    * **Router/Classifier:** 질문 의도가 '수치/통계'인지 '텍스트 정보'인지 분류.
    * **Text-to-SQL (RDB 접근):** * 모델: `gpt-oss-120b` (SQL 생성 및 결과 포맷팅에 최적화)
        * 대상: 국내/해외 주가 테이블, ETF 상품 마스터, 재무제표, 컨센서스 테이블
        * **필수 지침:** "ETF 수익률 등 수치 데이터는 반드시 SQL을 통해 DB에서 실시간 값을 가져올 것."
    * **RAG (VDB 접근):** 뉴스, 리포트 등이 필요할 때 호출.
    * **Markdown Formatter:** 조회된 DB 결과(Table)를 사용자가 보기 편한 Markdown 표 형식으로 변환하여 출력.

### B. 마켓 센싱 (Market Sensing)
* **패턴:** VDB(뉴스, SNS) 검색 -> `gpt-oss-120b` 요약 및 감성 분석

### C. 상관관계 분석 (Correlation Analysis)
* **패턴:** RDB 데이터 추출(SQL) -> Python Code Interpreter -> 상관계수 산출 및 시각화


## 4. 인프라 아키텍처 및 네트워크 토폴로지 (Infrastructure & Network)
*참고: '인프라 아키텍처 1안 - 리서치 센터 사용자' 도면 기준*

### A. 클러스터 구성 (Kubernetes Cluster Strategy)
1.  **운영 클러스터 (Operation Cluster):**
    * **Master Node:** API Gateway 역할을 수행하며 외부(AP서버) 요청을 수신.
    * **Worker Node:** GPU(H200) 및 CPU 자원이 배치된 실제 연산 노드.
2.  **네임스페이스 분리 (Namespace Isolation):**
    * **Namespace A (AI 플랫폼):** 전사 공용 AI 자원 (VDB, LLMOps 등).
    * **Namespace B (리서치센터 전용):** **[본 프로젝트의 핵심 영역]**
        * 배포 모듈: GenOS 플랫폼, 워크플로우 빌더, AI 서치, 관리자(Admin).
        * 특징: 물리적 GPU 자원은 공유하되, 논리적으로 격리된 리서치 센터 전용 Pod에서 구동됨.

### B. 통신 및 데이터 흐름 (Data Flow & Connectivity)
1.  **사용자 요청 흐름 (User Request Path):**
    * `내부망 PC` -> `L4 스위치` -> `리서치센터 AP 서버 (WAS)` -> `K8s Master Node (API 호출)` -> `GenOS (Namespace B)` -> `LLM/Workflow 수행`
    * *제약사항:* AP 서버는 클러스터 외부에서 Master Node를 통해 API로 통신함 (Direct Pod 접근 불가).

2.  **외부 데이터 반입 흐름 (External Data Ingestion):**
    * **소스:** 해외 재무/뉴스/공시, Dart, 금리/수출입 데이터.
    * **경로:** `대외계` -> `DMZ 구간 (API 중계서버/배치 연동)` -> `내부망 운영 클러스터` -> `통합 리서치 RDB / VDB 적재`
    * *보안정책:* 모든 외부 데이터는 **중계 서버(Relay Server)**를 거쳐 내부 RDB/VDB에 적재된 후 활용 가능 (실시간 인터넷 직접 접속 불가).

### C. 서버 및 스토리지 구성
* **리서치센터 전용 존:**
    * **WAS/AP 서버:** 웹 포털 및 비즈니스 로직 처리.
    * **통합 리서치 RDB:** 정형 데이터(주가, 재무, 거시경제) 저장소.
    * **NAS 서버:** 비정형 파일(PDF, 문서 등) 원본 저장 및 모델 학습용 데이터 공유.
